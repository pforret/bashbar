# PRP: API Usage SwiftBar Plugin

## Goal

Create `apiusage.1h.sh` — a SwiftBar plugin that shows AI API usage/spend in the macOS menu bar. The plugin auto-detects which providers are configured via API keys in `.env` and fetches available usage data for each.

## Why

- At-a-glance spend monitoring across multiple AI providers without opening dashboards
- Only shows providers the user actually uses (key-based auto-detection)
- Follows existing plugin patterns (core.sh, SF Symbols, .env)

## What

A plugin that cycles provider spend in the menu bar and shows a full breakdown in the dropdown. Each provider section shows spend, limits, or status depending on what the API supports.

### Success Criteria

- [ ] Plugin detects which API keys are set in `.env` and only queries those providers
- [ ] OpenRouter: shows spend in USD and credit limit
- [ ] OpenAI: shows spend via Admin API (or key-valid status if only regular key)
- [ ] Anthropic: shows key-active status (no billing API exists)
- [ ] Google Gemini: shows key-active status (no billing API exists)
- [ ] Graceful error handling: timeout, bad key, missing key all handled without crashing
- [ ] Uses `sf_metric`, `sf_detail_line`, `menu_line` from core.sh — no raw output
- [ ] Runs and produces valid SwiftBar output on macOS
- [ ] Configurable via `.env` / `.apiusage.env`

### API Research Results

#### OpenRouter — FULL SUPPORT

```bash
curl -s "https://openrouter.ai/api/v1/auth/key" \
  -H "Authorization: Bearer ${OPENROUTER_API_KEY}"
```

Response:
```json
{
  "data": {
    "label": "my-key-name",
    "usage": 1.234,
    "limit": 50.0,
    "is_free_tier": false,
    "rate_limit": { "requests": 200, "interval": "10s" }
  }
}
```

- `data.usage` = cumulative spend in USD
- `data.limit` = credit limit in USD (null if no limit)
- `data.is_free_tier` = boolean
- Auth: `Authorization: Bearer $KEY`
- No org ID needed, lightweight endpoint

#### OpenAI — PARTIAL SUPPORT

The official Usage API requires an **Admin API key** (separate from inference keys):

```bash
# Requires OPENAI_ADMIN_KEY (not the regular OPENAI_API_KEY)
curl -s "https://api.openai.com/v1/organization/costs?start_time=$(date -v-30d +%s)&end_time=$(date +%s)&limit=1" \
  -H "Authorization: Bearer ${OPENAI_ADMIN_KEY}" \
  -H "Content-Type: application/json"
```

If only `OPENAI_API_KEY` is set (no admin key), we can only validate the key works:

```bash
curl -s "https://api.openai.com/v1/models" \
  -H "Authorization: Bearer ${OPENAI_API_KEY}" | head -c 100
```

A 200 response means the key is valid. No spend data available without admin key.

### Known Gotchas

```bash
# CRITICAL: Only OpenRouter has a clean spend API. Others are limited.
# Anthropic and Google have NO billing API — only key validation possible.
# OpenAI billing requires a separate ADMIN key, not the regular API key.

# CRITICAL: curl on macOS may not have --connect-timeout, use --max-time instead.
# Always: curl --silent --max-time 10 ... 2>/dev/null

# CRITICAL: JSON parsing in bash without jq is fragile.
# Use grep/sed/awk for simple field extraction, but handle missing fields.
# Alternative: check if jq is available and prefer it.

# GOTCHA: .env variables may contain quotes — bash source handles this natively.
# GOTCHA: API keys can contain special chars — always double-quote: "${VAR}"
```

### Tasks

```yaml
Task 1: Create apiusage.1h.sh with boilerplate
  - Copy structure from template.1h.sh
  - Set metadata: title "AI API Usage", desc, deps (curl)
  - Source core.sh
  - Define .env defaults (all keys empty by default)
  - Stub empty plugin_output()
  - Test: ./apiusage.1h.sh help

Task 2: Implement JSON field extraction helper
  - Create json_field() function to extract a value from JSON by key
  - Strategy: try jq first (if installed), fall back to grep+sed
  - Must handle: nested "data" objects, null values, missing fields
  - Pseudocode:
      json_field() {
        local json="${1}" field="${2}"
        if command -v jq &>/dev/null; then
          echo "${json}" | jq -r ".${field} // empty"
        else
          # grep/sed fallback for simple flat fields
          echo "${json}" | grep -o "\"${field}\":[^,}]*" | head -1 | sed 's/.*://' | tr -d '" '
        fi
      }

Task 3: Implement provider check functions
  - check_openrouter(): curl /api/v1/auth/key, extract usage+limit
  - check_openai(): if ADMIN_KEY set, curl /v1/organization/costs; else curl /v1/models for validation
  - check_anthropic(): curl /v1/models with x-api-key header, check HTTP status
  - check_google(): curl /v1beta/models?key=, check HTTP status
  - Each function prints a status line that plugin_output() can parse
  - Pattern: echo "OK|$spend|$limit|$detail" or "ERROR|$message"

Task 4: Implement plugin_output()
  - Detect which keys are set: [[ -n "${OPENROUTER_API_KEY:-}" ]]
  - For each configured provider, call its check function
  - Parse result and call sf_metric/sf_detail_line appropriately
  - Providers with spend data: sf_metric "dollarsign.circle" "$X.XX" "Provider: $X.XX" "green"
  - Providers with status only: sf_metric "checkmark.circle.fill" "Provider" "Provider: active" "green"
  - Providers with errors: sf_metric "exclamationmark.triangle" "Provider" "Provider: error" "orange"
  - If NO providers configured: sf_metric "key.fill" "No keys" "No API keys configured"
  - Add menu_line links to each provider's dashboard

Task 5: Add .env examples and test
  - Add commented API key examples to .env
  - Test with no keys: should show "no API keys configured"
  - Test with OPENROUTER_API_KEY set (if user has one)
  - Test with invalid key: should show error gracefully
  - Test: ./apiusage.1h.sh (full output)
  - Test: ./apiusage.1h.sh help
```

### Pseudocode for plugin_output()

```bash
plugin_output() {
  local has_any=false

  # OpenRouter
  if [[ -n "${OPENROUTER_API_KEY:-}" ]]; then
    has_any=true
    local or_json or_usage or_limit
    or_json="$(curl --silent --max-time 10 "https://openrouter.ai/api/v1/auth/key" \
      -H "Authorization: Bearer ${OPENROUTER_API_KEY}" 2>/dev/null)"
    if [[ -n "${or_json}" ]] && [[ "${or_json}" != *"error"* ]]; then
      or_usage="$(json_field "${or_json}" "data.usage")"
      or_limit="$(json_field "${or_json}" "data.limit")"
      # CRITICAL: format as $X.XX — usage is already in USD
      sf_metric "dollarsign.circle" "OR \$${or_usage}" "OpenRouter: \$${or_usage}" "green"
      sf_detail_line "creditcard" "Spend: \$${or_usage}"
      [[ -n "${or_limit}" ]] && [[ "${or_limit}" != "null" ]] && \
        sf_detail_line "chart.bar" "Limit: \$${or_limit}"
    else
      sf_metric "exclamationmark.triangle" "OR err" "OpenRouter: error" "orange"
    fi
    menu_line "OpenRouter Dashboard | href=https://openrouter.ai/activity"
  fi

  # OpenAI — similar pattern, check OPENAI_ADMIN_KEY first, fall back to OPENAI_API_KEY
  # Anthropic — key validation only
  # Google — key validation only

  if [[ "${has_any}" == false ]]; then
    sf_metric "key.fill" "No keys" "No API keys configured in .env"
    detail_line "Add API keys to .env or .apiusage.env"
  fi
}
```

## Anti-Patterns to Avoid

- Do NOT raw `echo` for SwiftBar output — use core.sh functions only
- Do NOT require all API keys — only query configured ones
- Do NOT assume jq is installed — provide grep/sed fallback
- Do NOT hardcode API keys — always from .env
- Do NOT make the plugin slow — use `--max-time 10` and consider parallel curl
- Do NOT show provider sections for unconfigured providers in the dropdown
- Do NOT write directly to `_swiftbar_headers` or `_swiftbar_dropdown`
